  ---
  Section 1: Problem & Motivation (reframed)

  The v1 design led with the academic citation (Huang et al.) and authorship
  bias. The v2 reframes:

  Lead with the practical constraint: Within implementation-process, a single
  session handles implement → test → review → fix. The agent that wrote the code
   is the only one present to review it. Starting a separate session for review
  means pausing, spawning, reviewing, passing findings back, resuming — terrible
   UX.

  Sub-agents are the mechanism: Claude Code's Agent tool spawns sub-agents that
  inherently lack parent conversation history. This is a platform behavior, not
  a design achievement. The design's contribution is choosing to exploit this
  for review quality and curating what context the sub-agent receives.

  Supporting evidence (secondary): Cite the Huang et al. research as
  corroborating evidence for why self-review is problematic, not as the primary
  justification.

  Inspiration: The approach was inspired by the cove-isolated pattern but
  adapted significantly — CoVe does per-question factual verification while this
   does per-domain quality evaluation.

  Does this reframing look right before I continue?

● Section 2: Core Principle & Isolation Boundary (updated)

  Core principle stays the same — isolation is about authorship and session
  context, not knowledge. Sub-agents get full design rationale. The bias comes
  from living through implementation, not from knowing why decisions were made.

  Isolation boundary table — one addition: The v1 table excluded "alternatives
  considered but not taken" from sub-agents. Review v2 points out that knowing
  "approach X was tried and caused regression Y" prevents reviewers from
  suggesting the exact same broken approach. So:

  - New column in "Given to sub-agent": Curated rejected approaches summary —
  brief notes on what was tried and why it failed, without the full debugging
  narrative
  - This is NOT the implementation session's false starts. It's a deliberate,
  curated summary the main agent prepares before spawning sub-agents (part of
  the artifact preparation step)

  Framing adjustment: Acknowledge that Agent tool sub-agents inherently lack
  parent conversation history — the platform provides the isolation mechanism.
  The design's contribution is the curation: deciding which artifacts to pass
  (spec, diff, checklists, rejected approaches) and which to withhold (session
  narrative, debugging context).

● Section 4: Detailed Flows — Isolated Code Review (major rework)

  Still a three-phase pipeline, but Phase 2-3 change fundamentally.

  Phase 1 — Isolated Detection (parallel) — mostly unchanged

  Two independent reviewers spawn simultaneously:
  - Sub-agent A (code-reviewer agent): Same as v1 — receives git diff, spec
  context, language checklists, capy read access, plus curated rejected
  approaches summary. Produces P0-P3 findings.
  - Sub-agent B (pal codereview): Same as v1 — external model, naturally
  isolated. Change: output stays in native format, not force-mapped to P0-P3.

  Both run in parallel.

  Phase 2 — Annotation (replaces "Reconciliation")

  The main agent receives both finding sets and performs annotation, not 
  judgment:

  1. Duplicate merging: When both reviewers flag the same logical issue (matched
   by file location + issue description), merge into one entry. Tag as
  "corroborated" — independent confirmation. Severity stays as assessed; if they
   disagree on severity, show both assessments.
  2. Author context annotations: For any finding where the main agent's session
  context is relevant, add a clearly-labeled annotation explaining the
  implementation decision. This is context, not a disposition — "I chose X
  because Y" not "this finding is invalid."
  3. Author-sourced findings: If the close re-reading triggers new observations,
   the main agent MAY add them, clearly tagged as "author-sourced". Distinct
  from sub-agent findings so the user can weight accordingly.
  4. pal follow-up (optional): If a pal finding is ambiguous, the main agent MAY
   use pal's follow-up interaction to clarify before presenting to the user.

  What's gone: No disposition categories. No "Confirmed/Disputed" judgments. No
  "main agent cannot add findings" restriction. No severity escalation on
  agreement.

  Phase 3 — Presentation (simplified)

  Single report to user, organized by agreement level:
  1. Corroborated findings (both reviewers flagged) — highest signal
  2. Single-reviewer findings (code-reviewer only or pal only) — evaluate
  individually
  3. Author-sourced findings (main agent added during annotation) — clearly
  separated

  Each finding shows: reviewer attribution, severity, the finding itself, and
  any author context annotation. The pal section uses pal's native format; the
  code-reviewer section uses P0-P3.

  Does this flow work?

● Section 6: Error Handling (new section, absent from v1)

  This is modeled on cove-isolated's error handling pattern.

  Code review (solid-code-review:isolated):

  ┌──────────────────────────┬──────────────────────────────────────────────┐
  │         Failure          │                   Behavior                   │
  ├──────────────────────────┼──────────────────────────────────────────────┤
  │                          │ Present pal findings only. Note the failure. │
  │ code-reviewer sub-agent  │  Suggest standard review                     │
  │ times out or fails       │ (/kk:solid-code-review) as supplement for a  │
  │                          │ second opinion.                              │
  ├──────────────────────────┼──────────────────────────────────────────────┤
  │ pal unavailable          │                                              │
  │ (listmodels returns      │ Present code-reviewer findings only. Note    │
  │ nothing, or codereview   │ that external model review was unavailable.  │
  │ call fails)              │                                              │
  ├──────────────────────────┼──────────────────────────────────────────────┤
  │                          │ Abort isolated mode. Display message         │
  │ Both reviewers fail      │ suggesting fallback to /kk:solid-code-review │
  │                          │  (standard mode).                            │
  ├──────────────────────────┼──────────────────────────────────────────────┤
  │ Sub-agent produces       │ Attempt best-effort parsing. If completely   │
  │ malformed output         │ unparseable, treat as failure (apply rules   │
  │                          │ above).                                      │
  └──────────────────────────┴──────────────────────────────────────────────┘

  Spec review (implementation-review:isolated):

  ┌─────────────────────────┬───────────────────────────────────────────────┐
  │         Failure         │                   Behavior                    │
  ├─────────────────────────┼───────────────────────────────────────────────┤
  │ spec-reviewer sub-agent │ Abort isolated mode. Suggest fallback to      │
  │  times out or fails     │ /kk:implementation-review (standard mode).    │
  ├─────────────────────────┼───────────────────────────────────────────────┤
  │ Sub-agent produces      │ Same: best-effort parse, then failure         │
  │ malformed output        │ fallback.                                     │
  └─────────────────────────┴───────────────────────────────────────────────┘

  Large diffs: The sub-agent handles batching internally. If the diff exceeds
  the sub-agent's context window, the spawning workflow should note the
  limitation and suggest the user scope the review to specific files or tasks.

  Section 7: Invariants (revised)

  Down from four to three:

  1. No silent drops — every sub-agent finding must appear in the report
  (unchanged)
  2. Author-sourced findings must be tagged — if the main agent adds findings
  during annotation, they're clearly labeled as "author-sourced" (new, replaces 
  the old "no new findings" ban)
  3. Agreement = confidence, not severity — corroborated findings get a tag, not
   a severity bump. If reviewers disagree on severity, show both. (replaces the 
  old severity escalation rule)

  Removed: "Main agent cannot add new findings" (invariant #2 from v1)
  Removed: "Agreement increases effective severity by one level" (invariant #4
  from v1)

  Section 8: Validation Plan (new section)

  After shipping, run both modes (standard and isolated) on 5-10 real task
  reviews. Compare:
  - Unique findings per mode — does isolated surface issues standard misses?
  - False positive rate — what fraction of findings does the user reject?
  - User acceptance rate — what fraction does the user act on?
  - Corroboration rate — how often do both reviewers flag the same issue?
  (measures reviewer independence)

  Not a formal experiment — a "Validation Plan" section with these questions and
   a commitment to evaluate after ~10 uses.

  Section 9: Updated Design Decisions

  1. Opt-in, not default — (unchanged) isolated review costs more
  2. Two reviewers for code, one for spec — (unchanged) code quality benefits
  from multiple perspectives
  3. Rationale is context, not contamination — (unchanged) sub-agents get full
  design rationale
  4. User reconciles, agent annotates — (new, replaces "main agent reconciles")
  the agent provides implementation context but doesn't judge findings. The user
   makes the final call.
  5. No silent drops — (unchanged) every finding reaches the user
  6. Native formats preserved — (new) pal output stays in its own format rather
  than being force-mapped to P0-P3
  7. Session-level isolation flag — (new) chosen once at implementation-process
  start, applies to all review checkpoints
