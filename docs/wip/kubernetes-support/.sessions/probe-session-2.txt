===========================================================
FINDINGS
===========================================================

Substitution is triggered specifically by the ${...} brace syntax applied to
CLAUDE_PLUGIN_ROOT, regardless of markdown container.

Forms that DO NOT prevent substitution (all still substituted):
  - inline code backticks
  - fenced code blocks (plain, bash, markdown, tilde)
  - indented code blocks (4-space)
  - blockquotes
  - HTML comments
  - backslash / double-backslash escapes before the dollar

Forms that DO survive unsubstituted:
  - PROBE_A: bare $CLAUDE_PLUGIN_ROOT (no braces at all)
  - PROBE_G: HTML entity for the dollar: &#36;{CLAUDE_PLUGIN_ROOT}

Conclusion: the harness is matching the literal string "${CLAUDE_PLUGIN_ROOT}"
(with the dollar-brace-name-brace shape) and doing a text replacement. It is
NOT aware of markdown block types, so putting the variable inside code fences
does not protect it. Breaking the $->{ adjacency (via bare form, or by turning
the $ into &#36;) is the only thing that escapes substitution.

===========================================================
RECOMMENDATION
===========================================================

For CLAUDE.md / ADR / SKILL prose that needs to reference the variable
literally, use: bare $CLAUDE_PLUGIN_ROOT (no braces).

It is the simplest form that survives unsubstituted in every container tested
(prose, inline backticks, code fences, blockquotes, indented code, HTML
comments). If you MUST keep brace syntax for shell-correctness reasons, use
&#36;{CLAUDE_PLUGIN_ROOT} in prose — but note this is a markdown entity and
will render as ${CLAUDE_PLUGIN_ROOT} to human readers only after markdown
rendering; inside raw-text contexts (code fences) it stays as &#36;.
