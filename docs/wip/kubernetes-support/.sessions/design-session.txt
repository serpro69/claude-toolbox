  Question 2 of N — Detection trigger

  Now we need to decide how review-code activates the K8s profile on a given
  diff. This matters because it sets the false-positive rate (YAML is
  everywhere) and affects how the same detection logic is reused by test,
  document, and implement.

  - A. Content signature only. Read YAML files, match on top-level apiVersion: +
   kind: (manifests), apiVersion: v2 in Chart.yaml (Helm), or filename
  kustomization.yaml (Kustomize). Accurate; slower (opens every YAML).
  - B. Path heuristic only. Files under conventional dirs: k8s/, manifests/,
  charts/, kustomize/, deploy/. Fast; brittle (depends on project convention).
  - C. Combined, content as authority. Path heuristic as a fast pre-filter,
  content signature as the deciding check, filename fallback
  (kustomization.yaml, Chart.yaml). Most robust; slightly more code to
  implement.
  - D. Explicit opt-in only. User passes a flag or adds profile: k8s to
  tasks.md. No automatic detection. Zero false positives; forgetful by default.

  My recommendation: C. Reasoning: (1) path heuristics are cheap, so using them
  as a pre-filter makes the common case free; (2) content signature prevents
  false positives on non-K8s YAML (CI configs, app configs, OpenAPI specs) and
  false negatives on non-conventional layouts (deploy/overlays/prod/); (3)
  kustomization.yaml has no apiVersion/kind at the root and must be caught by
  filename, so the "filename fallback" isn't optional. D is too friction-heavy
  given your stated intent is a frictionless flow.
