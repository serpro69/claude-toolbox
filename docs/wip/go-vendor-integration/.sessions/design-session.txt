  - Con: The two systems don't talk. samber's security rules fire independently,

      as: design-patterns.md

  Section 2: Vendor script behavior

  The vendor script lives at scripts/vendor-profiles.sh. It takes one argument:
  the path to a manifest file. Invoked as make vendor-go (which calls
  scripts/vendor-profiles.sh scripts/go-vendor-manifest.yml).

  Execution flow:

  1. Parse the manifest. Read each upstream entry. For each, resolve the GitHub
  raw content base URL from repo and ref (e.g.,
  https://raw.githubusercontent.com/samber/cc-skills-golang/v1.1.3/).
  2. Fetch each source file. Download via curl from the raw URL. Fail loud (exit
   1) if any fetch returns non-200.
  3. Apply the keep transform. Per-file keep overrides the upstream's
  keep_default. Three modes:
    - all вҖ” verbatim copy (used for references/ files)
    - from_first_h1 вҖ” strip everything before the first line matching ^# . Exit
  1 if no H1 found (fail loud, don't produce garbage).
    - headings: [list] вҖ” extract only named H2 sections (escape hatch, unlikely
  to be used initially)
  4. Rewrite relative links. Scan for markdown links pointing to files within
  the same skill directory (e.g., [see injection](references/injection.md)). If
  the link target is another vendored file, rewrite to its as filename. If not
  vendored, strip the link (leave text, remove URL). This prevents broken links.
  5. Write to target. Place the transformed content at
  klaude-plugin/profiles/go/<phase>/<as>. Create the phase directory if it
  doesn't exist.
  6. Update index.md. For each phase directory that received files, regenerate
  the content between injection markers (<!-- BEGIN VENDORED --> / <!-- END 
  VENDORED -->). Always-load entries (no condition) get a markdown link +
  one-line description. Conditional entries get the link + description + **Load 
  if:** <condition>. Content outside the markers is untouched.
  7. Report. Print a summary: files fetched, transforms applied, phases updated.

  Section 6: Developer workflow and future extensibility

  Running the vendor tool:

  make vendor-go          # fetch, transform, place, update index.md files
  make test-structure     # validate bidirectional invariant (also runs at end 
  of vendor-go)

  First-time setup: go mod tidy pulls in the YAML parsing dependency (likely
  gopkg.in/yaml.v3, already in the Go ecosystem вҖ” no exotic deps). No other
  setup needed.

  Updating when samber releases a new version:

  1. Edit scripts/go-vendor-manifest.yml вҖ” bump ref: v1.1.3 to the new tag
  2. Run make vendor-go
  3. Review the diff вҖ” the changed files are the vendored content under
  klaude-plugin/profiles/go/
  4. Commit

  If samber restructures files (renames, moves), the fetch step fails loud (404
  on the old path). The developer updates the source paths in the manifest and
  re-runs.

  Adding a new samber skill to the profile:

  1. Add a files entry to the manifest вҖ” specify source, phase, as, and optional
   condition
  2. Run make vendor-go
  3. The tool places the file and updates the index.md markers
  4. Commit

  Adding a second upstream:

  Append a new entry to the manifest array with its own repo, ref, keep_default,
   and files. The tool iterates all upstreams in order. No code changes needed.

  Adding vendor support for another profile (e.g., Python):

  Create scripts/python-vendor-manifest.yml, add a make vendor-python target
  pointing at it. The vendor tool is profile-agnostic вҖ” the manifest specifies
  the target profile's phase directories. The Go tool itself needs no changes.
